Операции, которые будут выполняться часто:
- Поиск объектов недвижимости
- Поиск пользователей (убираем)
- Поиск записей на осмотр здания / пользователя

### Стратегии кеширования

Рекомендации для ваших сценариев
Для всех трех сценариев базовой стратегией взаимодействия с хранилищем (Redis/Memcached) лучше всего подходит Cache-Aside. Он наиболее предсказуем, прост в отладке и стандартен для веб-API. Однако нюансы реализации различаются.
1. Поиск объектов недвижимости (фильтрация)
Проблема: Высокая кардинальность параметров (цена, город, паттерны). Ключей может быть бесконечно много, что приведет к вымыванию полезных данных из кеша (cache pollution).
Решение:
Используйте Cache-Aside.
Ключ кеша: Хеш от параметров фильтрации (например, sha256("min=100&max=500&city=Moscow")).
Оптимизация (Session ID): Идея с "параметром сессии" хороша, но реализуется иначе. Сервер не должен хранить состояние сессии с результатами поиска (это масштабируется плохо). Вместо этого:
Фронтенд отправляет запрос с фильтрами.
Бэкенд формирует ключ кеша из фильтров.
Если есть в кеше — возвращаем.
Если нет — идем в БД, кладем в кеш с TTL (например, 5-10 минут, так как цены могут меняться).
Важно: Не используйте бесконечный TTL. Для поиска лучше короткий TTL, чем сложная инвалидация.
2. Поиск пользователей по логину
Проблема: Низкая кардинальность, данные меняются редко (имя, email), но критичны для безопасности/актуальности.
Решение:
Используйте Cache-Aside.
Ключ кеша: user:login:{login}.
Инвалидация: При обновлении профиля пользователя обязательно удаляйте ключ из кеша (invalidate), а не обновляйте его. Это гарантирует, что следующий read получит свежие данные из БД.
TTL можно поставить побольше (часы), так как данные стабильны.
3. Поиск записей на осмотр (по ID объекта/пользователя)
Проблема: Данные динамические (новые записи добавляются, старые отменяются). Частые чтения одного и того же ID.
Решение:
Используйте Cache-Aside.
Ключ кеша: inspections:user:{user_id} или inspections:object:{object_id}.
Инвалидация: Самая важная часть. При создании/удалении/изменении записи на осмотр вы должны удалить соответствующий ключ из кеша.
Если удаления нет, данные устареют. Write-Through здесь не нужен, так как запись идет через API, которое контролирует и БД, и кеш.

### TTL для кеша

Поиск объектов недвижимости (Фильтрация)
TTL: 300 – 900 секунд (5–15 минут).
Обоснование:
Цены и доступность меняются не каждую секунду.
Пользователь, подбирающий квартиру, не ожидает, что цена изменится прямо во время ввода фильтров.
Высокая вариативность запросов: если TTL будет слишком большим, кеш быстро забьется "мусорными" ключами от уникальных комбинаций фильтров, которые больше не повторятся. Короткий TTL позволяет этим ключам естественным образом устаревать.
Совет: Если нагрузка очень высокая, можно использовать Stale-While-Revalidate: отдавать устаревшие данные из кеша, пока в фоне обновляете их для следующего запроса.
2. Поиск пользователей по логину
TTL: 3600 – 86400 секунд (1–24 часа).
Обоснование:
Данные пользователя (имя, аватар, email) меняются крайне редко.
Ключей мало (низкая кардинальность), они всегда "горячие".
Даже если пользователь сменил имя, задержка в 1 час критичной не будет для большинства систем (кроме случаев немедленной блокировки/бана, но это обычно решается через отдельный флаг в кеше с коротким TTL или через инвалидацию при изменении).
Важно: Обязательно используйте Invalidate (удаление ключа) при обновлении профиля. TTL здесь работает как страховка на случай, если код инвалидации сработает некорректно.
3. Записи на осмотр (Inspections)
TTL: 60 – 300 секунд (1–5 минут).
Обоснование:
Это самые динамичные данные. Пользователь может записаться на осмотр, и другой пользователь должен увидеть, что слот занят, почти сразу.
Частота чтения высокая (все смотрят расписание конкретного объекта), поэтому кеш эффективен.
Риск устаревших данных высок: если кто-то занял слот, а кеш хранит старую информацию 10 минут, это приведет к конфликту бронирования.
Стратегия: Здесь TTL вторичен. Главное — жесткая инвалидация при любом изменении (CREATE/UPDATE/DELETE записи). TTL нужен только как защита от "вечного" кеша, если событие изменения было пропущено.

## Rate Limit

1. Выбор стратегии
Для API-сервисов, особенно с учетом требований к безопасности (авторизация) и пользовательскому опыту (поиск), лучшим выбором является Sliding Window Counter (или Sliding Window Log, если нужна высокая точность).
Почему не другие:
Fixed Window: Уязвим к "boundary attacks". Если лимит 100 req/min, злоумышленник может отправить 100 запросов в 59-ю секунду первой минуты и 100 запросов в 1-ю секунду второй минуты. Итого 200 запросов за 2 секунды.
Token Bucket / Leaking Bucket: Хороши для сглаживания трафика (rate shaping), но сложнее в реализации точных лимитов на короткие промежутки времени (например, "не более 5 попыток входа в минуту"). Они лучше подходят для ограничения пропускной способности (bandwidth), а не количества действий.
Рекомендация: Используйте Sliding Window Counter.
Он сочетает точность Sliding Window Log с эффективностью памяти Fixed Window.
В Redis это реализуется через сортированные множества (Sorted Sets) или атомарные инкременты с expiration.
Библиотеки: slowapi (для FastAPI) часто использует алгоритмы на базе окон

### Эндпоинты

Риск: Брутфорс паролей, Credential Stuffing.
Стратегия: Жесткое ограничение по IP и/или по логину (email/phone).
Лимит:
5–10 попыток в минуту на один IP.
3–5 попыток в минуту на один аккаунт (логин).
Дополнительно: После 3 неудачных попыток можно требовать CAPTCHA или увеличивать задержку ответа (exponential backoff).
Почему так мало: Подбор пароля — операция дешевая для атакующего, но критичная для вас. 5 попыток в минуту дают 300 попыток в час. Этого достаточно для легитимного пользователя, который забыл пароль, но слишком мало для эффективного брутфорса простого пароля.

Комбинированный (рекомендуется для авторизации):
Считать лимиты и по IP, и по логину. Если превышен любой из них — блокировать.
rate:login:ip:{client_ip} AND rate:login:account:{login}
