# Проектирование документной модели

## Основные сущности

У разработанной в прошлых лабораторных работаах системы по управлению недвижимостью
есть следующие сущности:

- `User` - сущность, описывающая пользователя системы. Она может представлять как
  клиента, так и продавца недвижимости.

- `Property` - сущность, описывающая объект недвижимости, который продается.

- `Viewing` - сущность, описывающая запись на просмотр объекта недвижимости.

- `Address` - сущность, описывающая местоположение целевого объекта недвижимости.

### Определение атрибутов

Для описания каждой из перечисленных выше сущностей, с которыми оперирует система,
предлагается использовать следующие атрибуты:

- `User`
  - `login` - логин пользователя. Можно представить в виде строкового типа данных.
  - `first_name` - имя пользователя. Можно представить в виде строкового типа данных.
  - `last_name` - фамилия пользователя. Можно представить в виде строкового типа данных.

- `Property`
  - `price` - цена объекта недвижимости. Можно представить в виде целочисленного типа,
    описывающее стоимость в целевой валюте. При этом значение должно быть неотрицательным.

  - `status` - статус объекта недвижимости. Так как объект может быть только в двух
    состояниях (продан, или вытсавлен на продажу), то можно использовать enum.

- `Viewing`
  - `viewing_date` - время просмотра объекта недвижимости. Можно представить в виде
    типа данных, описывающих дату.

- `Address`
  - `city` - город, где находится объект недвижимости. Можно представить в виде строкового типа данных.
  - `country`- страна, в которой находится объект недвижимости. Можно представить в виде строкового типа данных.
  - `street`- улица в городе, на которой находится объект недвижимости. Можно представить в виде строкового типа данных.
  - `building`- номер объекта недвижимости на улице. Можно представить в виде целочисленного типа данных.
    При этом значение должно быть строго больше единицы, так как обычно так принято нумеровать здания на улицах.
  - `apartment` - номер объекта в здании, если объект недвижимости является квартирой. В противном случае эта информация
    может не храниться в объекте. Можно представить в виде целочисленного типа данных, который при этом строго больше нуля.

### Взаимосвязи между сущностями

Между сущностями можно провести следующие связи:

- `owner` - связь между `Property` и `User`. Описывает связь владельца объекта недвижимости.
- `property` - связь между `Property` и `Address`. Описывает связь между адресом и объектом недвижимости.
  При этом эта связь `1:1`, так как делаем предположение, что у всех объектов уникальные адреса.
- `user` - связь между `Viewing` и `User`. Описывает связь, кто записался на осмотр объект.
- `property` - связь между `Viewing` и `Property`. Описывает связь, на осмотр какого объекта назначена запись.

### Определение коллекций

Предлагается для каждой из перечисленных выше сущностей организовать данные в отдельные коллекции.
Исключением является только `Address`. Так как в действительно эти значения неразрывны с объектом недвижимости
(часто, когда нам надо оперировать с объектом недвижимости, нам надо оперировать и с его адресом),
то выгоднее использовать Embedded Documents, и сохранять информацию об адресе внутри объекта недвижимости.

Перечисленные выше взаимосвязи между сущностями предлагается представить в виде механизма ссылок.
